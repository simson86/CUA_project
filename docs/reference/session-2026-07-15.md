## 3. ⚠ API 응답시간(지연) 측정의 한계 — 명확히 함
`timed_create`(`bench_multiturn.py:105`)는 `time.perf_counter()`로 `client.create()` 왕복을
잰다. **방법론은 건전**(단조시계, 스크린샷 캡처는 타이머 밖으로 제외됨). 하지만:
- 이 값은 **"서버 순수 연산"이 아니라 왕복 지연** = 네트워크(한국↔구글) + 서버추론 + SDK파싱.
  → **네트워크가 나쁘면 숫자도 그대로 나빠짐.** 랩탑의 그때 망 상태가 통째로 들어감.
- **재시도가 타이머 안에 있음**(`call_with_retry` 포함): 연결 끊겨 백오프 `sleep(2→4→…)` +
  실패 시도까지 합산 → 그 턴 폭증. 재시도 걸린 턴 latency는 순수 API지연 아님.
- 표의 "API지연(s)"은 턴별이 아니라 **작업당 합계**(턴 수에 비례). 호출 1회 속도는 `--per-turn`.
- **결론:** 같은 세션·같은 망 안의 **상대 비교**로만 유효. 서로 다른 날/장소 절대비교 금지.
- (개선 여지, 미적용) 재시도 시간 타이머 밖으로, 워밍업 1회 버리기, 턴별 중앙값.

## 4. 데이터-사용량 벤치와 그 함정 (핵심)
작업 "내 한달간 데이터 사용량이 궁금해" × minimal/low/medium × 2회 실측
(`multiturn-data-2026-07-15.md`). 표면상 medium이 2턴·최저비용으로 압도했지만,
**두 교란이 겹쳐 그 결론은 못 쓴다:**

1. **소스 오류(정답 축 무효):** 모두 **기기 설정 데이터 카운터**(`11.89 GB`)를 읽음.
   의도한 정답은 **통신사 청구서(청구 주기) 기준**. 기기 카운터 ≠ 청구서. → 청구서 기준이면
   **6개 run 전원 실패.** "정답 도달"이 아니라 "아무 숫자 도달"이었음.
2. **순서·상태 이월:** 실행 순서 minimal→low→medium 고정 + `reset_home()`은 **HOME만**
   (앱 force-stop 안 함). medium(마지막)은 앞 run들이 **데워놓은 기기**에서 돌아 유리.
   "2턴 직행"이 사고력이 아니라 워밍업 인공물일 수 있음.

**모델 학습설은 기각:** 추론은 가중치 불변, run끼리 대화맥락 단절(`previous_interaction_id`는
한 작업 내 체이닝만). 원인은 모델 기억이 아니라 **기기 상태 이월**.
