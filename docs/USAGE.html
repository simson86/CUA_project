CUA_run 사용법

Gemini Computer Use 기반 안드로이드 자동화 — 라이브 에이전트 & 벤치마크 실행 가이드
목차
0. 사전 준비 1. 라이브 에이전트 (main.py) 2. bench_thinking.py (첫 판단) 3. bench_multiturn.py (완주) 4. 턴별 상세 (--per-turn) 5. 지표 읽는 법 6. 사고 수준 설정 7. 팁 & 주의 8. 측정 결과 문서
⚠ 공통 필수 — 파이썬은 반드시 py로 실행하세요. 이 PC의 python은 msys2라 패키지가 없어 ModuleNotFoundError가 납니다. 모든 명령은 repo 루트에서 실행합니다.

0. 사전 준비

  • 기기 연결: adb devices에 폰이 device로 표시되어야 함 (USB 디버깅 허용).
  • API 키: .env에 GEMINI_API_KEY=... (템플릿은 .env.example).
  • 패키지: py에 google-genai, python-dotenv 설치돼 있음.
# 연결 확인
adb devices
# 없으면 SDK 경로의 adb 직접 호출 (Windows)
& "$env:LOCALAPPDATA\Android\Sdk\platform-tools\adb.exe" devices

1. 라이브 에이전트 — live/main.py

목표를 주면 실제로 작업을 수행합니다 (캡처→판단→실행 반복). 한글 입력은 ADBKeyboard로 처리되고 종료 시 원래 키보드로 복원됩니다.

py live/main.py "설정 앱을 열어 다크 모드를 켜"
py live/main.py "유튜브에서 노래 틀어줘"
# 인자 없으면 기본값 "Open the Settings app"
py live/main.py

모놀리식 버전도 있음: py mobile_agent/agent.py (실행 후 작업을 물어봄).

2. 첫 판단 벤치 — tools/bench_thinking.py

"현재 화면에서 다음 액션 하나"를 정하는 한 번의 호출만 재서 사고 수준별 지연·토큰·비용을 비교합니다. 작업을 실제로 수행하진 않아 안전하고 빠릅니다.

py tools/bench_thinking.py --levels minimal,low,medium,high --runs 3
# 여러 명령 파일로, 결과 저장까지
py tools/bench_thinking.py --tasks-file task.txt --out docs/reference/th.md
# 기기 없이 저장된 스크린샷으로
py tools/bench_thinking.py --image shot.png --levels low,high

출력 예

[minimal] run 1/3:   8.36s  in=1170 out=41 thought=0
[low    ] run 1/3:   9.29s  in=1170 out=26 thought=99
... 이어서 수준별 평균 표(지연 min~max, 총tok, 비용)

옵션

옵션기본설명
--task "…"설정 다크모드측정할 목표 문장
--tasks-file—명령 목록 파일(줄마다 하나, 앞 번호 자동 제거)
--levelsminimal,low,medium,high잴 사고 수준들(쉼표)
--runs3수준마다 반복(평균)
--image—이 PNG로 측정(기기 불필요)
--out—결과 마크다운 저장 경로
--price-in / --price-outPRICING1M 토큰당 USD 단가 덮어쓰기

3. 완주 벤치 — tools/bench_multiturn.py

작업을 끝까지 수행시키며 수준별 완주율·턴/액션·에러·API지연·벽시계·총토큰·비용을 잽니다. 실제로 기기를 여러 스텝 조작하고 오래 걸립니다. 안전·되돌릴 수 있는 작업만 쓰세요(구매·결제·전송 금지).

py tools/bench_multiturn.py --tasks-file task.txt --levels minimal,low,medium --runs 2 --max-turns 15
# 단일 작업 한 번
py tools/bench_multiturn.py --task "삼성 노트에 메모 만들어줘" --levels low --runs 1
# 결과 저장
py tools/bench_multiturn.py --tasks-file task.txt --out docs/reference/mt.md

옵션

옵션기본설명
--task "…"—단일 작업 (없으면 내장 안전작업 3종)
--tasks-file—작업 목록 파일 (앞 번호 자동 제거)
--levelsminimal,low,medium,high잴 사고 수준들
--runs2작업·수준마다 반복
--max-turns12완주 못 하면 이 턴에서 중단(미완 처리)
--per-turnoff턴마다 API지연·토큰·액션·시간을 실시간 출력 (§4)
--out—결과 마크다운 저장(작업마다 중간 저장)
--price-in / --price-outPRICING단가 덮어쓰기

최종 스크린샷은 %TEMP%\cua_multiturn_shots에 t{작업}_{수준}_run{n}.png로 저장 → 실제 성공을 눈으로 검증하는 용도.

4. 턴별 상세 — --per-turn

턴마다 그 호출의 API 지연·토큰과 실행한 액션(이름·인자)·실행 시간을 그때그때 출력합니다.

py tools/bench_multiturn.py --task "설정 앱 열어" --levels low --runs 1 --per-turn

=== 작업 1/1 [low]: 설정 앱 열어 ===
      턴1: API  10.3s | in=1165 out=28 thought=32 총=1225
        → list_apps {} [0.13s]
      턴2: API   7.6s | in=3958 out=34 thought=25 총=4017
        → open_app {'package_name': 'com.android.settings'} [0.78s]
      턴3: API   7.3s | in=5107 out=10 thought=16 총=5133
  run 1: 완주 턴2 액션2 에러0 | API 25.1s / 벽시계 29.8s | tok ... 총=10375 | $0.003432
  • 턴N: API …s | in/out/thought/총 → 그 턴 모델 호출의 지연·토큰
  • → open_app {…} [0.78s] → 그 턴이 낸 액션과 ADB 실행 시간
  • 입력 토큰이 턴마다 커지는 건 서버가 히스토리를 누적하기 때문(previous_interaction_id).

5. 지표 읽는 법

지표뜻
API 지연완주까지 순수 모델 호출 시간 합 (진짜 latency, 네트워크·재시도 포함)
벽시계액션 실행 + 화면 안정 대기(SETTLE 0.6s)까지 포함한 작업당 총 시간
총tok완주까지 전 턴의 토큰 합(입력이 턴마다 누적 → 총량이 커짐)
사고tok모델이 생각에 쓴 토큰. 수준↑이면 증가(비용 동인)
완주율모델이 max-turns 안에 done을 선언한 비율 (자기 판단)
⚠ 완주 선언 ≠ 정답. 모델이 엉뚱한 화면에서 done을 선언하며 오답을 낼 수 있습니다 (실측에서 "장바구니 비었음" 오답 사례 확인). 반드시 최종 스크린샷/답변으로 실제 성공을 검증하세요.

6. 사고 수준(thinking level) 설정

값: minimal · low · medium · high. 벤치는 --levels로 지정하고, 라이브 에이전트는 .env의 CU_THINKING_LEVEL로 전체에 적용됩니다.

# .env
CU_THINKING_LEVEL=low   # 비우면 모델 기본

실측 요약: 평이한 작업은 low가 무난(턴·비용 최소, 헤맴 없음). 자세한 근거는 아래 결과 문서 참고.

7. 팁 & 주의

  • 이 세션에서 바로 돌리려면 프롬프트 맨 앞에 !를 붙이면 됩니다: !py tools/bench_thinking.py --levels minimal,high --runs 2
  • 비용 $의 단가는 placeholder입니다(PRICING). 토큰량은 실측이니 공식 가격표로 단가만 갱신하면 정확해집니다.
  • 기기 없이 토큰·지연만 볼 땐 bench_thinking.py --image를 쓰세요(입력 토큰은 화면이 달라도 ~1170으로 거의 일정).
  • 완주 벤치는 오래 걸리므로 여러 작업이면 백그라운드로 돌리는 게 편합니다. 재시도·중간저장이 내장돼 슬립/끊김에 강합니다.
  • 노트/파일을 만드는 작업은 반복 실행 시 결과물이 누적됩니다(사후 정리 필요).
